
# phoenixd.NET

## Project Overview

**phoenixd.NET** is a .NET 10 client and integration library for [phoenixd](https://phoenix.acinq.co/server),
the self-custodial Bitcoin/Lightning server daemon from [ACINQ](https://acinq.co). It wraps the full
phoenixd HTTP API, maintains a resilient websocket connection for real-time payment events, and
bridges those events to browser clients over **SignalR** — making it easy to add Lightning payments
to Blazor (or any ASP.NET Core) application.

> Targets **.NET 10** and tracks **phoenixd v0.8.0** (lightning-kmp 1.12.0, Taproot channels).

## Features

- **Complete API coverage** — node/channel management, invoices, BOLT12 offers, Lightning
  addresses, LNURL (pay/withdraw/auth), on-chain sends, invoice/offer decoding, liquidity-fee
  estimation, fee bumping, and paginated payment history.
- **Resilient websocket client** — automatic reconnection with exponential backoff, multi-frame
  message reassembly, graceful cancellation, and `wss://` support, exposing both raw and strongly
  typed (`PaymentReceived`) events.
- **Real-time SignalR bridge** — forwards incoming payments to the originating browser client.
- **Webhook signature verification** — `PhoenixdWebhookValidator` validates phoenixd's
  `X-Phoenix-Signature` (HMAC-SHA256) so you can trust incoming webhook callbacks.
- **Typed error handling** — non-success responses surface as `PhoenixdApiException` carrying the
  status code and raw body.
- **Docker support** for building and running phoenixd itself.

## Prerequisites

- .NET 10 SDK
- A running phoenixd instance (see [Running phoenixd in Docker](#running-phoenixd-in-docker))

## Setup

### 1. Configuration

phoenixd authenticates with HTTP Basic auth. The password is generated by the daemon on first run
and stored in `~/.phoenix/phoenix.conf` (`http-password`). A read-only password
(`http-password-limited-access`) is also generated. Add a `PhoenixConfig` section to your
`appsettings.json`:

```json
"PhoenixConfig": {
  "Token": "<http-password from phoenix.conf>",
  "LimitedAccessToken": "<http-password-limited-access (optional)>",
  "Host": "http://127.0.0.1:9740",
  "Username": "phoenix",
  "WebhookSecret": "<webhook-secret from phoenix.conf (optional)>",
  "RequestTimeoutSeconds": 30
}
```

| Setting                 | Description                                                                 |
| ----------------------- | --------------------------------------------------------------------------- |
| `Token`                 | Full-access HTTP password. Required for payments and reads.                 |
| `LimitedAccessToken`    | Optional read/receive-only password.                                        |
| `Host`                  | Base URL of the phoenixd HTTP API (default port **9740**).                  |
| `Username`              | Basic-auth username (phoenixd ignores it, but one is still sent).           |
| `WebhookSecret`         | Optional; used to verify incoming webhook signatures.                       |
| `RequestTimeoutSeconds` | HTTP request timeout applied to the API client (default 30).                |

### 2. Register the services

```csharp
using Phoenixd.NET;
using Phoenixd.NET.Hubs;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSignalR();                                 // required for the payment bridge
builder.Services.ConfigurePhoenixdServices(builder.Configuration);

var app = builder.Build();

app.MapHub<PaymentHub>("/paymentHub");                         // SignalR hub endpoint
app.Run();
```

`ConfigurePhoenixdServices` registers:

- `INodeService` and `IPaymentService` as typed `HttpClient`s,
- the singleton `PhoenixdClient` websocket listener,
- a hosted `PhoenixdClientBackgroundService` that runs the listener for the app's lifetime, and
- the `PhoenixdManagerService` that bridges payment events to SignalR.

> **Payments are not idempotent.** Do **not** attach an automatically-retrying resilience handler
> (e.g. `AddStandardResilienceHandler`) to the phoenixd HTTP clients: a retried `payinvoice`,
> `payoffer`, `paylnaddress` or `sendtoaddress` could result in a double payment. If you enable
> resilience globally via `ConfigureHttpClientDefaults`, exclude these clients.

### 3. Inject the services

You can inject `INodeService` / `IPaymentService` directly, or use `PhoenixdManagerService` which
exposes both plus the SignalR bridge.

```csharp
[ApiController]
[Route("api/payment-manager")]
public class PaymentManagerController(IPaymentService payments, INodeService node) : ControllerBase
{
    [HttpGet("balance")]
    public async Task<IActionResult> GetBalance(CancellationToken ct)
        => Ok(await node.GetBalance(ct));

    [HttpPost("receive-payment")]
    public async Task<IActionResult> Receive([FromBody] ReceiveLightningPaymentRequest req, CancellationToken ct)
        => Ok(await payments.ReceiveLightningPaymentAsync(
            req.Description, req.AmountSat, req.ExternalId, req.DescriptionHash, req.ExpirySeconds, req.WebhookUrl, ct));
}
```

## Real-time payments

### Server side

The convention is that a browser client sends its SignalR connection id as the invoice
`externalId`. When phoenixd reports the payment over the websocket, `PhoenixdManagerService`
forwards a `PaymentReceived` event to that exact client via the `"ReceivePayment"` message. You can
also subscribe to `PhoenixdManagerService.PaymentReceived` directly if you don't use SignalR.

### Blazor client

```csharp
_hubConnection = new HubConnectionBuilder()
    .WithUrl(Navigation.ToAbsoluteUri("/paymentHub"))
    .WithAutomaticReconnect()
    .Build();

string? connectionId = null;
_hubConnection.On<string>("ReceiveConnectionId", id => connectionId = id);   // use as externalId
_hubConnection.On<PaymentReceived>("ReceivePayment", payment => { /* update UI */ });

await _hubConnection.StartAsync();
```

> The hub emits the connection id on the `"ReceiveConnectionId"` message — make sure your client
> listens on exactly that name, otherwise `externalId` will be empty and notifications cannot be
> routed.

## Webhooks

In addition to (or instead of) the websocket, phoenixd can POST events to a webhook URL — either a
global URL configured at startup, or a per-invoice URL passed to `createinvoice`. Every request is
signed with `HMAC-SHA256` over the raw body using your `webhook-secret`, sent in the
`X-Phoenix-Signature` header. Always verify it before trusting the payload:

```csharp
app.MapPost("/phoenixd-webhook", async (HttpRequest request, PhoenixConfig config) =>
{
    using var reader = new StreamReader(request.Body);
    var body = await reader.ReadToEndAsync();
    var signature = request.Headers[PhoenixdWebhookValidator.SignatureHeaderName];

    if (!PhoenixdWebhookValidator.IsValid(config.WebhookSecret!, Encoding.UTF8.GetBytes(body), signature))
        return Results.Unauthorized();

    var evt = JsonSerializer.Deserialize<PaymentReceived>(body, PhoenixdJson.Default);
    // handle the payment...
    return Results.Ok();
});
```

## Running phoenixd in Docker

A Dockerfile that builds phoenixd from source is provided at
[`Phoenixd.NET/.docker/phoenixd/Dockerfile`](Phoenixd.NET/.docker/phoenixd/Dockerfile). It is pinned
to `PHOENIXD_BRANCH=v0.8.0`.

```bash
docker build -t phoenixd -f Phoenixd.NET/.docker/phoenixd/Dockerfile Phoenixd.NET/.docker/phoenixd
docker run -d -p 9740:9740 -v phoenixd-data:/phoenix/.phoenix --name phoenixd phoenixd
```

The daemon writes its config (including `http-password`) to the mounted `/phoenix/.phoenix` volume
on first run; copy the password into `PhoenixConfig:Token`.
